# BuildSmart 360 Security Status — Data Protection

This document lists the implementation status for all security requirements in the **Data Protection** category.

1. 🟢 **Already Implemented** — Only the minimum PII necessary for account registration and functionality must be collected. *(We collect only basic business profile metadata: company name, phone, and city. No high-risk individual PII or tax registration numbers are requested or saved.)*
2. ⚪ **Not Applicable** — PII displayed in the UI must be masked or partially hidden to reduce the risk of accidental exposure. *(Business info is only shown to the authenticated owner/builder themselves on their private settings panel; no customer support representatives or third parties have access.)*
3. 🟢 **Already Implemented** — Users must be able to permanently delete their account and all associated personal data from the system. *( Supabase delete cascade is fully operational; deleting an auth user instantly purges their profile, projects, and cost estimates.)*
4. ⚪ **Not Applicable** — Explicit user consent must be obtained before storing or using any cookies. *(We do not use tracking or advertising cookies. The application only stores necessary security session identifiers (JWT bearer tokens) in localStorage, which is exempt from cookie banners.)*
5. 🟢 **Already Implemented** — A privacy policy page/modal must be implemented and easily accessible within the application. *(Implemented a beautiful, fully-styled Privacy Policy & Terms modal accessible from the footer of the sign-in and registration forms.)*
6. ⚪ **Not Applicable** — Uploaded files must be stored outside the web root with restricted file system permissions. *(Imported rates CSV files are processed purely in RAM text streams and never written to the server's local disk; avatar images are sent directly to Supabase cloud storage buckets.)*
7. ⚪ **Not Applicable** — The web server must be configured to serve only explicitly allowed file types and extensions. *(Flask only acts as a REST API returning structured JSON responses or PDF binaries; it does not serve static file structures.)*
8. ⚪ **Not Applicable** — Sensitive data such as tokens, keys, and PII must be cleared from application memory after use to prevent exposure via memory dumps. *(Standard Python automatic garbage collection cleans up out-of-scope local variables dynamically in RAM.)*
9. 🟢 **Already Implemented** — Application caches must be cleared on logout so that sensitive data cannot be retrieved from cache after a session ends. *(Clicking logout deletes JWT tokens (bs_token) and profile caches from browser localStorage and memory.)*
10. ⚪ **Not Applicable** — Cloud storage resources such as S3 buckets must require authentication. No user-owned content must be publicly accessible without authorization. *(Managed natively via Supabase storage policies.)*
11. ⚪ **Not Applicable** — Swagger and OpenAPI documentation must be restricted from public access in production builds. *(We do not generate or expose Swagger documentation in production builds.)*
12. 🟢 **Already Implemented** — PII and session data must not appear in URL query parameters. Request body or headers must be used instead. *(Registration, login, and updates are passed securely via HTTP headers and request bodies.)*
13. ⚪ **Not Applicable** — Sensitive PII fields in the database must be encrypted at the application level, not relying solely on database-level encryption. *(We collect no high-risk financial, health, or personal identity numbers. Standard AES-256 block-level Transparent Data Encryption on Supabase databases is fully sufficient.)*
