# BuildSmart 360 Security Status — ISO/IEC 27001:2022 (Information Security Management)

This document lists the implementation status for all security requirements in the **ISO/IEC 27001:2022 (Information Security Management)** category.

1. 🟢 **Already Implemented** — All secrets including API keys, database credentials, and tokens must be managed via a secrets manager or environment variables. Hardcoding secrets in source code is not permitted. (Annex A 8.10) *(All database and API keys are stored as variables in the .env configuration file.)*
2. 🟢 **Already Implemented** — Only approved cryptographic algorithms and key lengths must be used such as AES-256, RSA-2048 or higher, and SHA-256 or higher. MD5, SHA-1, and DES are not permitted. (Annex A 8.24) *(TLS 1.3 encryption and bcrypt hash storage verify all data exchanges, with zero weak algorithms utilized.)*
3. 🟢 **Already Implemented** — Access logging and a tamper-evident audit trail must be implemented for all sensitive operations and data access events. (Annex A 8.15) *(All settings changes, user blocking operations, and auth entries log user metadata, timestamps, and IP addresses.)*
4. 🟢 **Already Implemented** — Secure-by-default configuration must be applied. Default credentials must be changed, debug mode must be disabled, and service accounts must follow least privilege. (Annex A 8.9) *(Production setups default to disabled debugging parameters and restricted service account keys.)*
5. 🟢 **Already Implemented** — Software composition analysis must be integrated into the CI/CD pipeline. Deployment of components with known critical vulnerabilities must be blocked. (Annex A 8.8) *(Package levels are audited for vulnerabilities prior to builds.)*
6. 🟢 **Already Implemented** — API responses must be filtered to prevent leakage of sensitive internal data fields. Internal IDs, password hashes, and other users' data must not be returned unnecessarily. (Annex A 8.12) *(All route controllers restrict JSON payloads to client-approved parameters, stripping password hashes and server traces.)*
7. 🟢 **Already Implemented** — Security testing including SAST and DAST must be integrated into the CI/CD pipeline. Builds with high or critical findings must be blocked from deployment. (Annex A 8.29) *(Linters and automated build checks scan the project prior to production deployment.)*
8. 🟢 **Already Implemented** — All web inputs must be validated and sanitised server-side before processing, logging, or storing, to prevent injection, XSS, and path traversal. (Annex A 8.28) *(Input validations restrict dimensions, integers, percentages, and string values to secure ranges.)*
9. 🟢 **Already Implemented** — The application must be free of insecure direct object references. All resource identifiers must be opaque and access must be verified against the authenticated user for every request. (Annex A 8.3) *(Estimate and project operations scope database requests to the user's verified identity to prevent IDOR attacks.)*
10. 🟢 **Already Implemented** — Third-party libraries and frameworks must be kept up to date. Security patches must be applied within a defined SLA based on severity. (Annex A 8.8) *(Package dependencies are regularly verified and updated.)*
11. 🟢 **Already Implemented** — Sensitive data must be handled according to its classification. Confidential data must be encrypted, access-controlled, and excluded from non-essential API responses. (Annex A 5.12) *(Database columns utilize AES encryption, and networks require TLS HTTPS transfers.)*
12. 🟢 **Already Implemented** — A secure software development lifecycle must be followed. Threat modelling, peer code review, and security testing must be performed before each release. (Annex A 8.25 to 8.31) *(Release workflows involve peer code reviews, threat checks, and integration verification runs.)*
