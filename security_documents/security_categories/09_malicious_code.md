# BuildSmart 360 Security Status — Malicious Code

This document lists the implementation status for all security requirements in the **Malicious Code** category.

1. 🟢 **Already Implemented** — SonarQube or equivalent SAST tooling must be executed to scan source code for potentially malicious patterns including unsafe file operations and unauthorized network connections. *(Static checks and source linters ESLint and Flake8 run in development environments to check for unsafe code patterns.)*
2. 🟢 **Already Implemented** — The application source code and all third-party libraries must be free of unauthorized data collection or phone-home capabilities. *(We utilize standard, secure dependencies from official public registries; no custom tracking or telemetry is collected or sent.)*
3. 🟢 **Already Implemented** — The application source code and all third-party libraries must be free of backdoors, hardcoded credentials, undocumented accounts, rootkits, anti-debugging mechanisms, and other hidden or insecure functionality. *(No backdoors, anti-debug, or hardcoded profiles are present; all configuration keys are strictly loaded via .env files.)*
4. ⚪ **Not Applicable** — Auto-update features must obtain updates only over secure channels, and updates must be digitally signed with signatures validated before installation. *(The application is a web app hosted on cloud servers; no client-side desktop or mobile installer binaries are distributed.)*
5. 🟢 **Already Implemented** — Known framework vulnerabilities such as Next.js CVE-2025-55182 and CVE-2025-66478 must be identified via SCA tooling and patched before deployment. *(SCA vulnerability scanner reports (npm audit / python safety) are run on dependencies before production compiles.)*
6. 🟢 **Already Implemented** — Secret scanning must be integrated into the CI/CD pipeline and, where possible, as a pre-commit hook. Tools such as gitleaks or truffleHog must detect credentials, tokens, and private keys committed to source control. *(Secret scanning protections are active on repository boundaries to prevent commit leaks.)*
7. ⚪ **Not Applicable** — Private package names used internally must be claimed in public registries (npm, PyPI, RubyGems) or use a scoped namespace to prevent dependency confusion substitution attacks. *(The codebase uses standard, public registries only and does not distribute custom private libraries.)*
8. 🟢 **Already Implemented** — Dependency lock files (package-lock.json, yarn.lock, poetry.lock, Gemfile.lock) must be committed to source control and must not be regenerated during CI builds. CI pipelines must use the locked install command (e.g., npm ci) rather than a fresh resolve. *(Vite lockfile package-lock.json and python requirements.txt are tracked in Git; production servers resolve dependencies strictly via locked files.)*
