# BuildSmart 360 Security Status — API and Web Service

This document lists the implementation status for all security requirements in the **API and Web Service** category.

1. 🟢 **Already Implemented** — All API endpoints must be secured with both authentication and authorization. *(All projects, estimates, and administrative endpoints enforce Bearer JWT authentication and builder authorization checks.)*
2. 🟢 **Already Implemented** — API responses must include only the data that the authenticated and authorized user is entitled to view. *(Scoping checks restrict project and estimate database queries using the caller's validated builder ID.)*
3. 🟢 **Already Implemented** — Message headers and payloads must be validated to ensure they have not been modified in transit. *(HTTPS/TLS automatically guarantees connection payload integrity in transit.)*
4. 🟢 **Already Implemented** — API response payloads must be trimmed to the minimum data required. Fields such as hashed passwords, internal IDs, or other users' data must not be returned to the client. *(All controllers map model rows to restricted DTOs, stripping password hashes and other builders' fields.)*
5. 🟢 **Already Implemented** — Hidden or undocumented endpoints must be removed from production JavaScript bundles and page source. *(Production client builds minified via Vite fully strip tree-shaken code and comments.)*
6. 🟢 **Already Implemented** — Server-side outbound HTTP requests must be restricted to an explicit allow list of destinations to prevent Blind SSRF. *(The server triggers no user-supplied outbound calls; all endpoints are hardcoded in environment configs.)*
7. 🟢 **Already Implemented** — Deprecated and unused API endpoints must be removed or access-protected in production to minimize the attack surface. *(No deprecated endpoints or API versions are active in the app.)*
8. ⚪ **Not Applicable** — GraphQL introspection must be disabled in production environments. Schema discovery must not be available to unauthenticated or unauthorized clients. *(The application utilizes REST API endpoints, not GraphQL.)*
9. ⚪ **Not Applicable** — GraphQL queries must enforce a maximum depth limit to prevent deeply nested queries from exhausting server resources. *(The application utilizes REST API endpoints, not GraphQL.)*
10. ⚪ **Not Applicable** — GraphQL query complexity and cost must be calculated and limited per request. Alias-based batching attacks must be rate-limited or disallowed. *(The application utilizes REST API endpoints, not GraphQL.)*
11. ⚪ **Not Applicable** — Incoming webhook payloads must be authenticated by validating an HMAC signature (e.g., X-Hub-Signature-256) using a shared secret before processing the request. *(The server exposes zero inbound webhook listener routes.)*
12. ⚪ **Not Applicable** — Webhook endpoints must prevent replay attacks by rejecting payloads with a timestamp outside a defined window (e.g., +/- 5 minutes) or by recording and checking nonces. *(The server exposes zero inbound webhook listener routes.)*
13. 🟢 **Already Implemented** — API endpoints must only accept and process explicitly allowlisted fields from request bodies. User-supplied JSON must not be deserialized directly into model objects without strict field filtering to prevent mass assignment. *(Request body values are parsed individually via dictionary get calls, preventing mass assignment attacks.)*
14. 🟢 **Already Implemented** — Every API endpoint that accesses a resource by ID must verify that the specific object belongs to (or is explicitly shared with) the requesting user. Authorization must be re-evaluated per resource on every request, not assumed from login state alone. *(Every estimate or project database operation validates ownership (builder_id == user_id) on each REST request.)*
15. 🟢 **Already Implemented** — API endpoints must enforce rate limits per authenticated user and per IP address, returning 429 Too Many Requests with Retry-After and X-RateLimit headers when limits are exceeded. *(Centralized rate limits are enforced via Flask-Limiter.)*
16. ⚪ **Not Applicable** — Collection endpoints that return lists of resources must enforce a maximum page size. Requests with a limit or page_size parameter exceeding the maximum must be capped server-side, not fulfilled in full. *(Our resources are small-scale metadata arrays specific to each builder. We do not support user-supplied pagination parameters that could cause resource exhaustion.)*
17. 🟢 **Already Implemented** — API endpoints must validate the Content-Type header on incoming requests. Requests with an unexpected or missing content type must be rejected with 415 Unsupported Media Type before the body is parsed or processed. *(An app-level before_request middleware hook enforces application/json Content-Type on all state-changing API endpoints, returning 415 on mismatch.)*
18. ⚪ **Not Applicable** — When an API version or endpoint is deprecated but not yet removed, it must return Sunset and Deprecation response headers so consumers can identify and migrate away before removal. *(No deprecated API versions exist.)*
19. 🟢 **Already Implemented** — When the application consumes third-party or internal service APIs, responses must be validated against an expected schema before the data is used, stored, or rendered. Data from external APIs must not be trusted as safe or correctly formed. *(External Supabase data is parsed and validated against expected structures before processing.)*
20. 🟢 **Already Implemented** — API endpoints must ignore X-HTTP-Method-Override headers and _method query parameters unless the application explicitly requires them. These must not be used to tunnel non-permitted HTTP methods through POST requests. *(Flask ignores these override structures by default since we do not install or register the HTTPMethodOverrideMiddleware.)*
