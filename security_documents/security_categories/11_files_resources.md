# BuildSmart 360 Security Status — Files and Resources

This document lists the implementation status for all security requirements in the **Files and Resources** category.

1. 🟢 **Already Implemented** — Uploaded files must be validated for type, size, and content before being accepted by the server. *(CSV uploads for rates master are checked for .csv extension extensions and parsed in-memory as structured text; avatar images are restricted at Supabase storage bucket boundaries.)*
2. 🟢 **Already Implemented** — Filenames provided by users must be sanitized or ignored entirely to prevent directory traversal and file overwrite attacks. *(We do not save uploaded files to local disk paths. The file stream is loaded directly in RAM text variables, preventing directory traverses or file overwrites.)*
3. ⚪ **Not Applicable** — User-supplied input that could be used to load external resources must be validated or ignored to prevent RFI and SSRF. *(The backend accepts zero user-supplied URLs or domains to connect to; all database and mail integrations are pre-configured.)*
4. 🟢 **Already Implemented** — Responses serving downloadable files must include the Content-Disposition: attachment header to prevent in-browser execution. *(Both Excel spreadsheet exports and PDF downloads utilize Flask's send_file(as_attachment=True) which attaches Content-Disposition headers, preventing in-browser HTML/JS executions.)*
5. 🟢 **Already Implemented** — User-controlled file metadata such as filenames and paths must never be passed directly to system APIs or shell commands. *(No shell commands or raw operating system subprocesses are executed.)*
6. 🟢 **Already Implemented** — Only code from known, trusted sources must be included or executed. Dynamic includes and remote file loading based on user input are not permitted. *(All imports are statically resolved; no dynamic evaluation code is present.)*
7. ⚪ **Not Applicable** — Uploaded files must be stored in directories that are not accessible by the web server and must have restricted permissions. *(No files are saved locally on the Flask server disk; they reside in RAM or securely inside cloud storage buckets.)*
8. 🟢 **Already Implemented** — The web tier must be configured to serve only files with explicitly permitted extensions to prevent information leakage. *(Web layers are configured to serve only specific response types: application/json, application/pdf, and Excel formats.)*
9. ⚪ **Not Applicable** — Uploaded files must never be served or executed as HTML or JavaScript content when accessed directly in the browser. *(We do not expose or serve raw uploaded file directories to the web.)*
10. ⚪ **Not Applicable** — Archive files (zip, tar, gz) must be checked for zip bomb attacks before extraction by enforcing a maximum uncompressed size limit and entry count. *(We do not accept or process archive files; only .csv tables are supported.)*
11. ⚪ **Not Applicable** — Archive extraction must validate that all extracted file paths resolve within the intended target directory. Paths containing ../ traversal sequences must be rejected before extraction to prevent zip slip attacks. *(We do not support archive extraction tasks.)*
