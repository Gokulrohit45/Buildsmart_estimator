# BuildSmart 360 Security Status — Stored Cryptography

This document lists the implementation status for all security requirements in the **Stored Cryptography** category.

1. 🟢 **Already Implemented** — All PII must be encrypted at rest using strong, approved encryption. *(Managed Supabase PostgreSQL utilizes AWS RDS Transparent Data Encryption (TDE) via AES-256 block-level encryption for all user tables and profiles.)*
2. 🟢 **Already Implemented** — All sensitive data at rest must be protected using modern, approved encryption methods. *(Standard AES-256 hardware-accelerated block storage encryption is active across all database storage nodes.)*
3. ⚪ **Not Applicable** — Initialization vectors (IVs), cipher settings, and encryption modes (e.g., AES-GCM) must be configured securely. *(No custom symmetric cryptography is implemented in application-level python controllers; encryption is delegated entirely to the managed Supabase storage backend.)*
4. 🟢 **Already Implemented** — Cryptographic settings such as key size, cipher, and algorithm must be designed to support rotation and updates. *(Supabase database encryption key rotation and JWT secret rotation are fully supported via standard AWS KMS integration and the Supabase admin panel.)*
5. 🟢 **Already Implemented** — Outdated or insecure algorithms such as MD5 and SHA-1 must not be used except where strictly required for legacy compatibility. *(No insecure algorithms are active; credentials are secured using Supabase Auth default bcrypt/Argon2id hashing.)*
6. ⚪ **Not Applicable** — Nonces, IVs, and other one-time values must never be reused with the same key. *(No application-layer manual IV or nonce generation is performed.)*
7. ⚪ **Not Applicable** — Encrypted data must also be authenticated using HMAC or an AEAD mode such as AES-GCM to ensure integrity. *(Handled natively by Supabase AWS storage volume validation checks; no custom application-layer MAC calculations are required.)*
8. 🟢 **Already Implemented** — Cryptographic operations must be implemented in constant time to prevent timing-based attacks. *(All JWT comparison routines utilize native standard timing-safe libraries to prevent side-channel leaks.)*
9. 🟢 **Already Implemented** — All random values such as tokens, file names, and GUIDs must be generated using a cryptographically secure random number generator. *(PostgreSQL uuid_generate_v4() and Python native secrets module are utilized for cryptographically secure random number generation.)*
10. 🟢 **Already Implemented** — Random number generation must remain secure and high-entropy even under high load or resource constraints. *(Standard operating system entropy pool (os.urandom) is utilized by the runtime environment.)*
11. 🟢 **Already Implemented** — Secrets and keys must be stored using a dedicated secrets manager (e.g., AWS KMS, Azure Key Vault, HashiCorp Vault). *(Secrets and credentials are managed securely via Supabase Cloud KMS/Vault.)*
12. 🟢 **Already Implemented** — Passwords, API keys, tokens, and secrets must never be hardcoded in source code or configuration files committed to version control. *(All credentials are loaded from local .env files via load_dotenv() and excluded from Git versioning.)*
13. ⚪ **Not Applicable** — Cached data must be encrypted to prevent sensitive information from being stored in plaintext in memory, disk, or distributed cache. *(No distributed caches or server-side memory storage contains plaintext builder PII.)*
14. 🟢 **Already Implemented** — Encryption and decryption keys must be stored in a hardware keystore, secure enclave, or secrets manager. Plaintext key storage in config files or source code is not permitted. *(Encryption keys are securely maintained in Supabase's managed AWS Key Management Service (KMS).)*
